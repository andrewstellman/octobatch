## Purpose

**Deming ("quality is built in, not inspected in"):** This document is the primary quality context
file for AI sessions working on Octobatch. Every coverage target, every scenario, and every
guardrail exists because we learned — expensively — what happens without it. Reading this file
first is not optional; it is how quality is built into every session before a line of code changes.

**Juran ("fitness for use"):** Octobatch's purpose is to orchestrate LLM pipelines reliably at scale.
"Fitness for use" means a 9,240-unit batch run completes correctly, recovers from crashes without
losing records, produces statistically valid output, and shows the operator accurate cost and status
information at all times. Tests passing is a floor, not a ceiling.

**Crosby ("quality is free"):** The scenario-to-test mapping in this file prevents the most expensive
bugs: the ones that pass all tests but silently corrupt data, inflate costs by 17x, or trap the
operator with no recovery path. Writing these tests costs hours. The bugs they prevent cost days.

## Coverage Targets

| Subsystem | Target | Why |
|-----------|--------|-----|
| `expression_evaluator.py` | 100% | The drunken sailor reseeding bug (`random.seed(unit_index)` → 77.5% bias) passed all visual inspection. Only statistical tests caught it. Every expression path must be covered because subtle seeding bugs produce plausible-but-wrong output. |
| `orchestrate.py` — state machine | 90% | Chunk state transitions are crash recovery. Untested transitions = silent attrition (1,693 records lost in the real incident). State machine completeness is non-negotiable. |
| `run_tools.py` — verify/repair | 90% | `verify_run()` and `repair_run()` are the safety net. If they have bugs, the silent attrition bug goes undetected. Every branch of their gap-detection logic must be tested. |
| `validator.py` | 90% | Schema + business logic validation is the only thing between LLM hallucinations and corrupted output. The trailing comma bug (11% failure rate) was only caught at scale. |
| `schema_validator.py` | 85% | Type coercion (Postel's Law) has subtle edge cases. The `wound_count` int/string confusion caused downstream failures. Every coercion path must be tested. |
| `config_validator.py` | 85% | Config validation errors fail silently in production runs. The 4-Point Link Rule gaps and expression step validation bypass (Scenario 9) were both config-layer bugs. |
| `generate_units.py` | 85% | Unit generation with repeat/seed logic is the upstream source. Seeding errors here corrupt every downstream step. |
| `realtime_provider.py` + `providers/` | 80% | Timeout handling and error categorization must be verified. The Gemini infinite hang (0% CPU, 40+ minutes) happened because the client had no timeout parameter. |
| TUI utilities (`runs.py`, `formatting.py`) | 90% | Pure functions with no UI dependencies. The hardcoded Flash pricing fallback (BUG-6: 17x cost undercount) is in this layer. Easy to test, no excuse not to. |
| TUI screens & widgets | Critical paths automated | Use `async with app.run_test() as pilot:` for keybindings and state transitions. Visual feel is in the Human Gate. |

## Coverage Theater Prevention

These test patterns look like tests but don't verify behavior. Flag them for removal:

- **Asserting a dict has keys** without checking values — `assert "unit_id" in result` when the function could return `{"unit_id": None}`
- **Asserting no exception was raised** without checking the return value — `process(fixture)` with no assertions
- **Asserting import succeeded** — `assert orchestrate is not None`
- **Asserting mocks return what they're configured to return** — mock returns X, test asserts result is X
- **Testing with data that can't trigger the bug** — verifying integrity on a perfect run, never testing with missing/orphaned/duplicated records
- **Counting output without checking values** — `assert len(results) == 5` when 5 wrong records would pass
- **Single-variant tests for cross-cutting properties** — testing seeded randomness only with Gemini, not all three providers

## Fitness-to-Purpose Scenarios

The scenarios in `ai_context/QUALITY.md` are authoritative. This section adds the automation layer:
test function names, requirement tags, and scenarios surfaced by code exploration beyond the
original ten.

### Scenario 1: Crash Recovery (The Silent Attrition Incident)
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 1]
**What happened:** A 9,240-unit batch run was interrupted mid-execution. The orchestrator recovered,
but 1,693 units were silently lost — they started but never completed, and `verify_run()` was not
yet implemented. Without verification, silent attrition is undetectable: the run reports "complete"
with N valid records while N−1,693 records simply vanish.
**The requirement:** `verify_run()` must detect every category of unit loss: missing (expected but
not in valid or failed), duplicated (appeared more than once in valid), orphaned (in valid but
not expected). `repair_run()` must create retryable chunks for every missing unit.
**How to verify:** `test_scenario_1_crash_recovery_verify_run_detects_missing_units()` —
create a run with 5 units where 2 are missing from all output files. Assert `verify_run()` reports
`total_missing == 2` and `integrity == "WARN"`.

### Scenario 2: Retry Granularity (The 4,300-Unit Cascade)
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 2]
**What happened:** 308 units failed across 64 chunks. Retry reset entire chunks rather than
individual failed units, reprocessing 4,300 units instead of 308 — a 14x waste multiplier.
**The requirement:** `retry_validation_failures()` must create retry chunks containing ONLY failed
units, not entire source chunks. A chunk with 5 failures out of 100 units produces a retry chunk
with exactly 5 units, not 100.
**How to verify:** `test_scenario_2_retry_granularity_only_failed_units_retried()` — create a
100-unit chunk with 5 failures. Assert retry chunk has exactly 5 units.

### Scenario 3: SIGINT Responsiveness
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 3]
**What happened:** Closing a laptop killed network connections. The orchestrator hung silently, then
crashed without logging. On restart, partial progress within a step was lost.
**The requirement:** `mark_run_paused()` must only set status to "paused" when not already terminal.
SIGINT handler must save manifest within 5 seconds. All API calls must have request-level timeouts.
**How to verify:** `test_scenario_3_sigint_pauses_non_terminal_run()` — assert `mark_run_paused()`
transitions "running" → "paused" and does NOT override "complete" or "failed".

### Scenario 4: Idempotent Extract (The 16,479 Duplicate Bug)
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 4]
**What happened:** `extract_units.py` appended to existing output on second run, doubling file count.
**The requirement:** Any extract/output operation must be idempotent. Running twice = running once.
**How to verify:** `test_scenario_4_verify_run_is_idempotent()` — verify_run on the same run_dir
twice returns identical results.

### Scenario 5: PID File Alignment
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 5]
**What happened:** PID stored in manifest from previous session; TUI checked manifest PID, found
dead process, declared "PROCESS LOST" even while new orchestrator ran correctly.
**The requirement:** `write_pid_file()` must write current PID to the filesystem. Manifest stores
PID only as a diagnostic backup, not as the source of truth for liveness checks.
**How to verify:** `test_scenario_5_write_pid_file_writes_current_pid()` — call `write_pid_file()`,
read the file, assert contents match `os.getpid()`.

### Scenario 6: Batch Mode Heartbeat
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 6]
**What happened:** Batch mode showed "Status: Running" then silence for 20-40 minutes. Operator
could not distinguish live polling from a hung process.
**The requirement:** Batch polling must emit heartbeat messages even when no state changes occur.
The 60-second heartbeat interval must be respected.
**How to verify:** `test_scenario_6_heartbeat_interval_config()` — assert `heartbeat_interval = 30`
constant exists in `orchestrate.py` and `_watch_loop()` uses it.

### Scenario 7: Statistical Correctness (The Drunken Sailor Bug)
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 7]
**What happened:** `random.seed(unit_index)` created correlated random streams. Sequential seeds
produce related sequences — the sailor landed in water 77.5% of the time instead of ~50%.
**The requirement:** `_repetition_seed` must be derived from `hash(unit_id)`, not from sequential
integers. Identical seeds must produce identical output. Different seeds must produce different output.
**How to verify:** `test_scenario_7_seeded_random_deterministic()` and
`test_scenario_7_different_seeds_produce_different_results()`.

### Scenario 8: Validation Catches LLM Errors
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 8]
**What happened:** 11% failure rate from trailing commas in JSON. Schema validator caught it, but
only after running 9,240 units. Type coercion allowed `wound_count` as string then failed downstream.
**The requirement:** Schema validator must handle trailing commas (via recovery), type coercion
(str→int for numbers), and must produce structured error output (not Python tracebacks).
**How to verify:** `test_scenario_8_schema_validator_handles_type_coercion()` and
`test_scenario_8_config_validates_type_constraints()`.

### Scenario 9: Expression Step Validation Enforcement
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 9]
**What happened:** Expression step with `validation:` entry in config bypassed validation entirely.
456 of 900 mathematically incorrect Blackjack hands flowed through unchecked because all AI advisors
assumed validation applied universally. TOOLKIT.md documented the wrong pattern. Three independent
models shared the same incorrect assumption.
**The requirement:** When an expression step has a `validation:` entry, the orchestrator must
run schema and business logic validation. `step_has_validation()` must return True for expression
steps with validation config. `run_expression_step()` must write failures to the failures file
when validation fails.
**How to verify:** `test_scenario_9_expression_step_validation_enforced()` — create a config with
an expression step that outputs `verification_passed: False`. Run validation. Assert unit appears
in failures file, not validated file.

### Scenario 10: Empty Retry Chunks Don't Block Completion
**Requirement tag:** [Req: formal — ai_context/QUALITY.md Scenario 10]
**What happened:** 1,000-repeat Blackjack run entered infinite loop after LLM step completed.
Retry chunks where ALL units were exhausted by expression fast-fail were stuck: orchestrator
tried to advance them, couldn't find previous step's validated.jsonl (0 units survived),
logged WARN, tried again next tick. 20 retry chunks stuck forever.
**The requirement:** When all units in a retry chunk are fast-failed by an expression step, the
chunk must be marked FAILED (terminal). The run must complete when all non-empty chunks finish.
The watch loop must not cycle indefinitely on empty chunks.
**How to verify:** `test_scenario_10_failed_chunks_no_retryable_work_marked_terminal()` — build
a manifest with a FAILED chunk that has `retries=0` (below max) but no failure files. The
watch_run startup logic must patch it to `retries=max_retries` (terminal).

### Scenario 11: "stuck" Status Blocks TUI Management
**Requirement tag:** [Req: formal — ai_context/DEVELOPMENT_CONTEXT.md BUG-2]
**What happened:** Rate-limited runs entered "stuck" status (sustained 429 errors). TUI status
guard only allows pause/kill on "running" status. Users cannot stop or manage stuck runs
through the TUI, requiring manual `kill` commands.
**The requirement:** TUI pause and kill handlers must accept "stuck" status in addition to "running".
Stuck runs have an active orchestrator PID that needs stopping — the same mechanism as running.
**How to verify:** `test_scenario_11_stuck_status_recognized_as_manageable()` — assert that
"stuck" is treated equivalently to "running" for pause/kill eligibility, or flag the missing
guard in TUI code.

### Scenario 12: "pending" Runs Not Resumable From TUI
**Requirement tag:** [Req: formal — ai_context/DEVELOPMENT_CONTEXT.md BUG-3]
**What happened:** Newly initialized runs (status: pending) cannot be resumed from the TUI R key.
Resume only works on paused/failed/zombie runs. Users must use CLI to start pending runs.
**The requirement:** TUI resume handler must detect "pending" status and launch the appropriate
CLI command (`--watch` or `--realtime`), not just re-attach to an existing process.
**How to verify:** `test_scenario_12_mark_run_running_allows_pending_to_running_transition()` —
test that the manifest status machine handles a pending-to-running transition correctly.

### Scenario 13: format_step_provider_tag Crashes on None Provider
**Requirement tag:** [Req: formal — ai_context/DEVELOPMENT_CONTEXT.md BUG-4]
**What happened:** `get_provider()` throws on missing API key, caught by broad except, leaves
`realtime_provider = None`. Then `format_step_provider_tag(realtime_provider)` dereferences None,
raising `AttributeError: 'NoneType' object has no attribute 'config'`.
**The requirement:** `format_step_provider_tag()` must handle None provider_instance gracefully,
returning "unknown" or a safe fallback string rather than crashing.
**How to verify:** `test_scenario_13_format_step_provider_tag_handles_none()` — call
`format_step_provider_tag(None)` and assert it returns a string, not an exception.

### Scenario 14: TUI Cost Calculation Uses Wrong Pricing Fallback (BUG-6)
**Requirement tag:** [Req: formal — ai_context/DEVELOPMENT_CONTEXT.md BUG-6]
**What happened:** `_calculate_cost_from_manifest()` falls back to `input_rate = 0.075, output_rate = 0.30`
(hardcoded Flash pricing) when `config.api.pricing` block is missing. Pricing is registry-driven,
so the fallback always triggers. Gemini 2.5 Pro runs were shown at $0.30/M (Flash batch)
instead of $5.00/M (Pro batch) — a 17x undercount. An $1,856 run appeared to cost ~$109.
**The requirement:** Cost calculation must load pricing from `scripts/providers/models.yaml` using
the manifest's provider+model. If the model isn't in the registry, show "cost unknown" — never
a wrong number. The registry is the single source of truth for pricing.
**How to verify:** `test_scenario_14_cost_calculation_uses_registry_not_hardcoded()` — assert that
`_calculate_cost_from_manifest()` does not contain hardcoded rate constants as a fallback path.

### Scenario 15: No Pre-Run Cost Estimate Before Fan-Out Commitment (BUG-7)
**Requirement tag:** [Req: formal — ai_context/DEVELOPMENT_CONTEXT.md BUG-7]
**What happened:** An Elena run estimated at ~$100 cost ~$1,856. The fan-out from 5,281 parent
units to 51,712 child units was the unaccounted multiplier. `--init` committed the user to the run
without showing cost estimates. `cost_cap_usd` only applies to realtime mode, not batch.
**The requirement:** Before starting any run that includes fan-out steps or significant unit
multiplication, the system must display the estimated total unit count and cost projection.
For `--init` in CLI: print estimate and require `--yes` or interactive confirmation.
**How to verify:** `test_scenario_15_init_run_without_yes_flag_requires_confirmation()` — verify
that the `--yes` flag exists in argument parser and that it gates some form of confirmation logic.

## AI Session Quality Discipline

1. **Read QUALITY.md before starting work.** Both `ai_context/QUALITY.md` and this file.
2. **Run the full test suite before marking any task complete:** `pytest tests/ quality/`
3. **Add tests for new functionality** — not just happy path, include error cases.
4. **If modifying state transitions:** add a test verifying the new path.
5. **If modifying validation:** add a test with deliberately malformed input.
6. **If modifying provider code:** add a test verifying timeout/error handling.
7. **Output a Quality Compliance Checklist before ending a session.**
8. **Never remove a scenario.** Only add new ones.

## The Human Gate

These checks require human judgment:

- TUI visual verification (stats populate, colors correct, Otto narrative makes sense)
- Operational feel (SIGINT stops within 5s, resume picks up correctly)
- Prompt template quality (spot-check generated content for domain correctness)
- Documentation accuracy (README steps work on clean machine)
- Cost estimate accuracy for novel pipelines (requires domain knowledge of token counts)
