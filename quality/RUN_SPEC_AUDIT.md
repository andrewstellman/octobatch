# Spec Audit Protocol: Octobatch

> Generated by Quality Playbook v1.2.0 on 2026-03-28.

## The Definitive Audit Prompt

Give this prompt **identically** to three independent AI tools (e.g., Claude, GPT-4o, Gemini). Each auditor works independently — no cross-contamination.

---

**Context files to read first:**

1. `CLAUDE.md` (root) — Architecture, code conventions, key design decisions
2. `ai_context/PROJECT_CONTEXT.md` — State machine, MANIFEST.json structure, chunk lifecycle, TUI screens
3. `ai_context/DEVELOPMENT_CONTEXT.md` — Active bugs BUG-1 through BUG-7, technical learnings
4. `quality/QUALITY.md` — 15 fitness-to-purpose scenarios (the cases most likely to reveal spec-code divergence)
5. `scripts/CONTEXT.md` — Orchestrator internals, function cross-reference
6. `scripts/tui/CONTEXT.md` — TUI application structure

**Source directories to read:**

- `scripts/orchestrate.py` — Main orchestrator (5000+ lines; focus on functions named in scrutiny areas)
- `scripts/run_tools.py` — verify_run(), repair_run(), generate_report()
- `scripts/expression_evaluator.py` — SeededRandom, evaluate_expressions()
- `scripts/config_validator.py` — validate_config()
- `scripts/tui/screens/home_screen.py` — TUI run list, key handlers
- `scripts/tui/screens/main_screen.py` — TUI cost calculation

**Task:** Act as the Tester. Read the actual code and compare it against the specifications in the context files above.

**Requirement confidence tiers:**

Requirements are tagged `[Req: tier — source]`. Weight findings by tier:
- **formal** — written by humans in a spec document. Authoritative. Divergence is a real finding.
- **user-confirmed** — stated by the user but not in a formal doc. Treat as authoritative.
- **inferred** — deduced from code behavior. Report divergence as NEEDS REVIEW, not a definitive defect.

**Rules:**

- ONLY list defects. Do not summarize what matches. Do not produce a "positive findings" section.
- For EVERY defect, cite specific file **and line number(s)**. If you cannot cite a line number, do not include the finding. Use `grep -n 'pattern' file` to find exact lines.
- **Grep before claiming missing.** Before saying a feature is absent, search the codebase. If found in a different file, that's PHANTOM, not MISSING.
- **Read function bodies, not just signatures.** Don't assume a function works based on its name. If a function is named `mark_run_complete()`, read whether it actually writes `status: complete` to the manifest.
- Classify each finding: **MISSING** / **DIVERGENT** / **UNDOCUMENTED** / **PHANTOM**
- For findings against inferred requirements, add: **NEEDS REVIEW**
- Known bugs BUG-1 through BUG-7 are documented in `ai_context/DEVELOPMENT_CONTEXT.md`. Do not re-report them unless you've found a different instance or root cause. Reference them as "BUG-N (known)" if you encounter them.

**Defect classifications:**

- **MISSING** — Spec requires it, code doesn't implement it
- **DIVERGENT** — Both spec and code address it, but they disagree on behavior
- **UNDOCUMENTED** — Code does it, spec doesn't mention it
- **PHANTOM** — Spec describes it, but it's implemented differently than described

**Project-specific scrutiny areas:**

> These are not optional. Read the named functions, at the specific line numbers, before answering each question.

**1. State machine completeness (`scripts/orchestrate.py`, `scripts/tui/screens/home_screen.py`)**

CLAUDE.md states: run status values are `running`, `paused`, `failed`, `complete`. Grep the codebase for all values assigned to `manifest["status"]` and all status values checked in conditional guards.

- Does `home_screen.py` handle the `"stuck"` status in its P (pause) and K (kill) key handlers?
- Does `home_screen.py` handle the `"pending"` status in its R (resume) key handler?
- Are there any status values that can be written to MANIFEST.json that are NOT handled by the TUI?

Cite the exact lines where each handler checks status, and list the statuses it includes and excludes.

**2. Expression step retry behavior (`scripts/orchestrate.py`)**

CLAUDE.md states: "Expression step validation failures are permanently failed (fast-fail), not retried, because re-running deterministic math on the same input produces the same result."

Find `retry_validation_failures()` (search for this function name). Read its body. Does it skip units from expression steps when building the retry chunk? If yes, cite the exact line where expression steps are excluded. If no, cite the exact line where they are (incorrectly) included.

**3. Seeded randomness correctness (`scripts/expression_evaluator.py`)**

CLAUDE.md states: "Seeded randomness uses `hash(unit_id + step_name) & 0x7FFFFFFF`, not sequential integers. Sequential seeds create correlated random streams."

Find `SeededRandom` and `create_seeded_interpreter()`. Read the actual seed derivation. Does it use `hash(unit_id + step_name) & 0x7FFFFFFF`? Does `SeededRandom` use its own private `_rng` instance (not the global `random` module)? Does it have a `max_iterations` guard on `loop_until` to prevent infinite loops?

Cite the exact line of the seed derivation expression and the `_rng` initialization.

**4. Retry chunk collision prevention (`scripts/orchestrate.py`)**

AGENTS.md states `get_next_retry_number()` was recently fixed to scan both disk directories AND manifest keys to prevent collision.

Find `get_next_retry_number()`. Read its body. Does it scan both:
  (a) on-disk directories matching `retry_*`, AND
  (b) manifest chunk keys matching `retry_`?

If either scan is missing, that's a DIVERGENT finding (CLAUDE.md says collision prevention is present; code doesn't implement both halves).

**5. Fan-out child ID format (`scripts/orchestrate.py`)**

CLAUDE.md states: "Child IDs: `{parent_id}__fan{NNN}`" where NNN is zero-padded 3 digits.

Find `run_fan_out_step()`. Read the child ID construction. Is the format exactly `{parent_id}__fan{NNN}` with 3-digit zero padding? Is `_fan_parent_id` and `_fan_index` added to each child record? Is the parent chunk marked terminal after fan-out?

Cite the exact line where child IDs are formatted.

**6. Cost calculation accuracy (`scripts/tui/screens/main_screen.py`)**

CLAUDE.md says pricing comes from `models.yaml`. Search `main_screen.py` for `_calculate_cost_from_manifest()` or similar cost functions. Does it load pricing from `models.yaml`, or does it use hardcoded fallback rates? Cite the exact line of any hardcoded rate values.

**7. `is_run_terminal()` dual-key lookup (`scripts/orchestrate.py`)**

CLAUDE.md states: "MANIFEST.json is the single source of truth." `is_run_terminal()` must handle both `retries` and `retry_count` keys in chunk metadata (historical format variation).

Find `is_run_terminal()`. Read its body. Does it check both `chunk.get("retries")` and `chunk.get("retry_count")`? If only one key is checked, chunks written with the other format will never be seen as exhausted — causing infinite retry loops.

**8. `repair_run()` context preservation (`scripts/run_tools.py`)**

`ai_context/DEVELOPMENT_CONTEXT.md` states repair_run() was recently fixed to preserve the richest context when selecting unit data: `len(record) > len(unit_data[uid])`.

Find `repair_run()`. Read its body. Does it use `len(record) > len(unit_data[uid])` (or equivalent) to select the richest record? Or does it use the first record found, or the last?

**9. `_watch_loop()` termination condition (`scripts/orchestrate.py`)**

Scenario 10 in `quality/QUALITY.md` describes an infinite loop when all FAILED chunks have no retryable work. Find `_watch_loop()`. Read its termination condition. Does the loop check `is_run_terminal()` before sleeping? Can it exit when all chunks are FAILED but none are retryable?

Cite the exact line of the termination check and what happens when `is_run_terminal()` returns True.

**10. SIGINT save timeout (`scripts/orchestrate.py`)**

AGENTS.md mentions `SIGINT_SAVE_TIMEOUT`. Find this constant. Find the SIGINT handler. Does the signal handler path actually use `SIGINT_SAVE_TIMEOUT` as a bounded timeout when saving state? Or does it save without a timeout (risking hang on disk I/O error during interrupt)?

## Running the Audit

1. Give the identical prompt above to **three independent AI tools** (suggested: Claude Sonnet, GPT-4o, Gemini 1.5 Pro or 2.0 Flash)
2. Each auditor works independently in a fresh session with no prior context
3. Provide all six context files (listed above) and access to the `scripts/` directory
4. Do not share one model's output with another before all three complete

## Triage Process

After all three models report, merge findings into `quality/spec_audits/YYYY-MM-DD-triage.md`:

| Confidence | Found By | Action |
|------------|----------|--------|
| Highest | All three | Almost certainly real — fix or update spec |
| High | Two of three | Likely real — verify and fix |
| Needs verification | One only | Could be real or hallucinated — deploy verification probe |

### The Verification Probe

When models disagree on factual claims, give one model the disputed claim and ask it to read the code and report ground truth. **Never resolve factual disputes by majority vote** — the majority can be wrong about what code actually does. The probe must cite a specific line number.

### Categorize Each Confirmed Finding

- **Spec bug** — Spec is wrong, code is fine → update spec document
- **Design decision** — Human judgment needed → open discussion before fixing
- **Real code bug** — Fix in small batches by subsystem
- **Documentation gap** — Feature exists but undocumented → update CLAUDE.md or CONTEXT.md
- **Missing test** — Code is correct but no test verifies it → add to `quality/test_functional.py`
- **Known bug** — Matches BUG-1 through BUG-7 in DEVELOPMENT_CONTEXT.md → confirm instance, record reference

## Fix Execution Rules

- Group fixes by subsystem, not by defect number
- Never one mega-prompt for all fixes
- Each batch: implement, test, have all three reviewers verify the diff
- At least two auditors must confirm fixes pass before marking complete
