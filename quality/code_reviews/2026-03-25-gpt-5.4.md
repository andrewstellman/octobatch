### scripts/orchestrate.py
- **Line 503:** `BUG` — `check_prerequisites()` ignores the effective provider override and can reject valid override-driven runs before they start.
  - **Expected:** When a run was initialized with CLI provider/model overrides, startup prerequisite checks should validate only the provider(s) that can actually execute after override resolution.
  - **Actual:** `check_prerequisites()` unions providers from the snapshotted config (`api.provider` plus per-step `provider`) and `watch_run()` / `realtime_run()` call it with config only, even though `init_run()` records `metadata.cli_provider_override` / `metadata.cli_model_override` in the manifest. A run forced to Gemini can still be blocked for missing `OPENAI_API_KEY` or `ANTHROPIC_API_KEY`.
  - **Why it matters:** This breaks the documented CLI-over-config precedence and threatens QUALITY Scenario 3 (`Provider Override Drift`) by making override test runs fail for the wrong reason.

- **Line 5032:** `BUG` — realtime retries do not honor `failure_stage`, so hard pipeline failures are retried as if they were recoverable validation failures.
  - **Expected:** Only `schema_validation` and `validation` failures should enter the retry path; `pipeline_internal` failures should remain terminal because retrying the same lost/malformed record does not fix the pipeline defect.
  - **Actual:** `run_realtime_retries()` reads every failure record and retries anything with `retry_count < max_retries`, without filtering on `failure_stage`.
  - **Why it matters:** This collapses the retryable vs. hard-failure boundary defined in `specs/VALIDATION.md` and threatens QUALITY Scenario 6 (`Retry Isolation Must Preserve Good Work`) by spending retries on failures the system has already classified as non-retryable.

- **Line 5472:** `BUG` — each new realtime invocation resets all persisted `retry_count` values back to zero, so retry exhaustion is not durable across resumes.
  - **Expected:** Failure records should keep their accumulated `retry_count` across `--realtime` restarts so max-attempt limits remain meaningful after interruptions and resumptions.
  - **Actual:** `realtime_run()` rewrites every `*_failures.jsonl` record with `failure["retry_count"] = 0` before processing, effectively granting a fresh retry budget on every invocation.
  - **Why it matters:** This defeats retry exhaustion and can turn bounded retries into unbounded spend/churn, directly threatening QUALITY Scenario 6 (`Retry Isolation Must Preserve Good Work`).

- **Line 4008:** `BUG` — the "no retryable failures remaining" terminalization guard treats any non-empty failures file as retryable work, so runs with only hard failures can stay non-terminal forever.
  - **Expected:** The guard should inspect failure records and only keep a `FAILED` chunk non-terminal when retryable stages (`schema_validation` / `validation`) are still present.
  - **Actual:** `watch_run()` marks `has_retryable = True` whenever a `*_failures.jsonl` file exists and has bytes; the duplicate logic in `realtime_run()` and the post-retry pass does the same. A chunk with only `pipeline_internal` failures therefore keeps `retries < max_retries`, but also has no code path left that can advance it.
  - **Why it matters:** This can leave the orchestrator polling forever even though no recoverable work remains, threatening QUALITY Scenario 1 (`Manifest Integrity After Crash`) and Scenario 6 (`Retry Isolation Must Preserve Good Work`).

- **Line 5299:** `BUG` — manifest failure counts are under-reported after realtime retries merge results back into the original chunk files.
  - **Expected:** After rewriting a chunk's failures file, `chunk_data["failed"]` should equal the full number of remaining failures for that chunk at that step, including hard failures and any retried units that still failed.
  - **Actual:** The code sets `chunk_data["failed"]` to the count of `remaining_failures` whose `unit_id` is in `retryable_failures`, which excludes hard failures and any non-retried records that were intentionally preserved in the same file.
  - **Why it matters:** This makes the manifest understate failure totals, which threatens QUALITY Scenario 1 (`Manifest Integrity After Crash`) and Scenario 6 (`Retry Isolation Must Preserve Good Work`) by making operator-visible state diverge from on-disk artifacts.
