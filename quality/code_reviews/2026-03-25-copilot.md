### scripts/orchestrate.py
- **Line 343:** `BUG` — Run status can be incorrectly changed from a terminal state.
  - **Expected:** A run that is `complete` or `failed` should not be changed back to `running` unless it was a premature completion and non-terminal chunks exist.
  - **Actual:** The `mark_run_running` function checks if all chunks are terminal. If so, it returns. However, it does not consider that the manifest's chunk list might be out of sync with the actual chunk directories on disk, for example if a retry chunk was created but the manifest write failed. A more robust check would be to scan the `chunks` directory on disk.
  - **Why it matters:** This threatens Scenario 1 (Durability and Correctness) and Scenario 6 (Operator Safety). An operator resuming a seemingly completed run could cause duplicate processing or unexpected costs.

- **Line 1088:** `BUG` — `retry_validation_failures` can re-process already validated units.
  - **Expected:** When retrying failures for a step, only the failed units should be re-processed. The input for the retry chunk should be the original input units that led to the failure.
  - **Actual:** The code looks for the input for the failed units in the previous step's `_validated.jsonl` file. However, it loads the entire file into `input_by_unit_id` and then filters. If a unit ID appears multiple times (e.g. due to a bug or complex fan-out), it might pick the wrong input. More importantly, the logic to build the retry chunk is complex and prone to error. A simpler approach would be to isolate failed units and their original inputs more directly.
  - **Why it matters:** This threatens Scenario 1 (Durability and Correctness). Re-processing successful units is inefficient and could lead to unintended side effects or increased costs.

- **Line 1158:** `BUG` — `retry_validation_failures` does not properly handle the case where the input source file for retries is missing.
  - **Expected:** If the input source file (e.g., `units.jsonl` or a previous step's `_validated.jsonl`) is missing, the retry logic should gracefully handle this, log an error, and not create a malformed retry chunk.
  - **Actual:** The code checks `if input_source.exists():` and if it does, it populates `input_by_unit_id`. If the file does not exist, `input_by_unit_id` remains empty. The code then proceeds to create a retry chunk, but any units that needed to be retried will be missing their input data, causing them to fail again in the retry chunk, but for a different reason (missing data).
  - **Why it matters:** This threatens Scenario 1 (Durability and Correctness). It can lead to silent data loss in retry chunks and permanent, unrecoverable failures for units that might have been retryable.
