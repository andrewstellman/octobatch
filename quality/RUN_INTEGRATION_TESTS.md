## Working Directory

All commands use **relative paths from the project root.** Run everything from the directory containing this file's parent (the project root). Do not `cd` to an absolute path or parent directory — use `./scripts/`, `./pipelines/`, `./quality/`, etc.

## Safety Constraints

- **DO NOT modify source code** during integration testing
- **DO NOT delete files** — if a test run fails, leave it for inspection
- **ONLY create files** in `runs/integration/` (see Pre-Flight)
- **If something fails, record it and move on** — the goal is observation, not debugging

## Pre-Flight Check

Before running integration tests, verify:

```bash
source .venv/bin/activate

# 1. Check API keys
[ -n "$GOOGLE_API_KEY" ] && echo "GOOGLE_API_KEY: set" || echo "GOOGLE_API_KEY: MISSING"
[ -n "$OPENAI_API_KEY" ] && echo "OPENAI_API_KEY: set" || echo "OPENAI_API_KEY: MISSING"
[ -n "$ANTHROPIC_API_KEY" ] && echo "ANTHROPIC_API_KEY: set" || echo "ANTHROPIC_API_KEY: MISSING"

# 2. Load .env if keys are missing
[ -f .env ] && set -a && source .env && set +a

# 3. Verify unit tests pass (safety net)
pytest tests/ -q --tb=no 2>&1 | tail -3

# 4. Create integration run directory
mkdir -p runs/integration

# 5. Validate pipeline configs before committing API calls
python3 scripts/orchestrate.py --validate-config --pipeline DrunkenSailor
python3 scripts/orchestrate.py --validate-config --pipeline Blackjack
python3 scripts/orchestrate.py --validate-config --pipeline NPCDialog
```

**Stop if any API key is missing.** The live tests will fail silently without it. Do not proceed to Steps 1–3 without all three keys.

**Stop if `--validate-config` fails.** Config errors detected locally won't be caught once the API calls start.

## Execution Commands

### Setup

```bash
source .venv/bin/activate
TS=$(date +%Y%m%d-%H%M%S)
echo "Session timestamp: $TS"
```

### Group 1 — Three providers in parallel (no provider collision)

```bash
# At most one run per provider simultaneously
python3 scripts/orchestrate.py --init --watch --pipeline DrunkenSailor \
  --run-dir runs/integration/ds-gemini-RT-$TS \
  --repeat 15 --realtime --provider gemini --yes --quiet &

python3 scripts/orchestrate.py --init --watch --pipeline Blackjack \
  --run-dir runs/integration/bj-openai-RT-$TS \
  --repeat 20 --realtime --provider openai --yes --quiet &

python3 scripts/orchestrate.py --init --watch --pipeline NPCDialog \
  --run-dir runs/integration/npc-anthropic-RT-$TS \
  --max-units 15 --realtime --provider anthropic --yes --quiet &

wait
echo "Group 1 complete"
```

## Post-Run Verification Checklist

For each completed run, verify at all 6 levels. Use `$RUN` as a placeholder for the run directory.

### Level 1 — Process (Did it exit cleanly?)

```bash
# Check log file for completion message
tail -5 $RUN/orchestrator.log | grep -i "complete\|finished\|done\|error"

# Check manifest status
python3 -c "import json; m=json.load(open('$RUN/MANIFEST.json')); print('status:', m['status'])"
```

**Pass:** status is `complete` (not `running`, `failed`, or `stuck`)

### Level 2 — State (Is the manifest terminal?)

```bash
python3 -c "
import json
m = json.load(open('$RUN/MANIFEST.json'))
chunks = m.get('chunks', {})
non_terminal = [(k,v) for k,v in chunks.items() if not any(v.get('state','').endswith(s) for s in ['VALIDATED','FAILED'])]
print(f'Run status: {m[\"status\"]}')
print(f'Non-terminal chunks: {len(non_terminal)}')
if non_terminal:
    print('  ', non_terminal[:3])
"
```

**Pass:** All chunks in terminal states (`*_VALIDATED` or `*_FAILED`), run status is `complete`.

### Level 3 — Data (Does output parse correctly?)

```bash
# Check validated output exists and is valid JSON Lines
STEP=$(python3 -c "import json; m=json.load(open('$RUN/MANIFEST.json')); print(list(m.get('pipeline',{}).get('steps',{}).keys())[-1])" 2>/dev/null || echo "analyze")
find $RUN -name "${STEP}_validated*.jsonl" | head -3 | while read f; do
  count=$(wc -l < "$f")
  python3 -c "
import json, sys
errors = []
for i, line in enumerate(open('$f')):
    try: json.loads(line)
    except: errors.append(i+1)
print(f'$f: {$count} records, {len(errors)} parse errors')
"
done
```

**Pass:** All `.jsonl` files parse without errors; record count matches expected unit count.

### Level 4 — Content (Do records have expected fields?)

Run the pipeline-specific checks below for the relevant pipeline.

#### DrunkenSailor content check

```bash
python3 -c "
import json
f = sorted(__import__('glob').glob('$RUN/*/analyze_validated*.jsonl'))[-1]
records = [json.loads(l) for l in open(f)]
required = {'scenario_name', 'final_position', 'steps_taken', 'path', 'outcome'}
missing_fields = [r for r in records if not required.issubset(r.keys())]
invalid_pos = [r for r in records if not (0 <= r.get('final_position', -1) <= 10)]
invalid_outcome = [r for r in records if r.get('outcome') not in ('fell_in_water', 'reached_ship')]
print(f'Records: {len(records)}')
print(f'Missing required fields: {len(missing_fields)}')
print(f'Invalid final_position (not 0-10): {len(invalid_pos)}')
print(f'Invalid outcome (not enum): {len(invalid_outcome)}')
# Distribution check
outcomes = [r.get('outcome') for r in records]
fell = outcomes.count('fell_in_water')
ship = outcomes.count('reached_ship')
print(f'Outcome distribution: fell_in_water={fell}, reached_ship={ship}')
print(f'Non-degenerate (both outcomes present): {fell > 0 and ship > 0}')
"
```

**Pass criteria:**
- 0 records with missing required fields
- All `final_position` values are integers in range 0–10
- All `outcome` values are exactly `"fell_in_water"` or `"reached_ship"` (no other values)
- Both outcomes appear at least once in 15 records (distribution non-degenerate)

#### Blackjack content check (play_hand step)

```bash
python3 -c "
import json
f = sorted(__import__('glob').glob('$RUN/*/play_hand_validated*.jsonl'))[-1]

```

**Pass criteria:**
- 0 records with missing required fields
- All `result` values are in `{player_wins, dealer_wins, push}`
- All `first_action` values are in `{hit, stand, double_down}`

#### NPCDialog content check (generate_dialog step)

**Pass criteria:**

## Known BUGs That May Surface During Integration Testing

These active bugs (from `ai_context/DEVELOPMENT_CONTEXT.md`) can affect integration test results:

| Bug | Symptom | Affected Tests |
|-----|---------|---------------|
| **BUG-4** | `format_step_provider_tag()` crashes when API key is missing | Any test where key check fails silently |
| **BUG-5** | No rate limit backoff — tight retry loop on 429 | High-load group runs; may see repeated 429 errors |
| **BUG-6** | Hardcoded pricing in TUI cost display (`output_rate=0.30`) | Cost shown in report may be 17× off vs. actual |

If tests fail due to BUG-5 (rate limiting in parallel groups), space out Group 1 runs manually: run each test sequentially or with a 30-second delay between starts.
