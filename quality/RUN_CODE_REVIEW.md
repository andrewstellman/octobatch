## What to Check

### Focus Area 1: Orchestrator State Machine (`scripts/orchestrate.py`)

**Where:** `mark_run_failed()`, `mark_run_paused()`, `mark_run_complete()`, `mark_run_running()`, `is_run_terminal()`, `_watch_loop()`, `tick_run()`

**What to check:**
- Every chunk state transition: are all states handled by the consumer? States: `{step}_PENDING`, `{step}_SUBMITTED`, `{step}_COMPLETE`, `{step}_VALIDATED`, `VALIDATED`, `FAILED`
- Run status: does every handler that checks status include ALL valid statuses? Known gap: `stuck` status is not handled in TUI handlers (BUG-2)
- `is_run_terminal()`: the `retries` vs `retry_count` dual-key lookup — is it consistent everywhere?
- `mark_run_running()`: the `all_terminal` guard prevents resuming a finished run — is it working?
- `_watch_loop()`: does the loop have a correct termination condition? Does it handle the case where all FAILED chunks have no retryable work? (Scenario 10)
- `SIGINT_SAVE_TIMEOUT`: is the save-on-interrupt path actually bounded by this constant?

**Why:** Untested state transitions are the root cause of the Silent Attrition Incident (1,693 records lost), the Infinite Loop bug (Scenario 10), and the PID misalignment bug (Scenario 5).

### Focus Area 2: Retry and Recovery Logic (`scripts/orchestrate.py`, `scripts/run_tools.py`)

**Where:** `retry_validation_failures()` (~line 892), `repair_run()`, `get_next_retry_number()`, `build_failures_map()`

**What to check:**
- `retry_validation_failures()`: are expression steps correctly skipped? (deterministic — retrying won't help)
- `retry_validation_failures()`: does it create retry chunks with ONLY failed units? (Scenario 2 requirement)
- `get_next_retry_number()`: does it scan BOTH disk directories AND manifest keys to prevent collision? (Recent bug fix — verify it's present)
- `repair_run()`: does it preserve the richest context when selecting unit data? (Recent bug fix — check `len(record) > len(unit_data[uid])` pattern)
- `build_failures_map()`: does it correctly exclude units that were later validated in retry chunks?

**Why:** Retry bugs produce either cascading reprocessing (4,300-unit cascade) or permanent loss (silent attrition). The retry chunk collision crash was a recent fix.

### Focus Area 3: Expression Step Execution (`scripts/orchestrate.py`, `scripts/expression_evaluator.py`)

**Where:** `run_expression_step()` (~line 4265), `is_expression_step()`, `step_has_validation()`, `evaluate_expressions()`, `SeededRandom`

**What to check:**
- Does `run_expression_step()` run validation when `step_has_validation()` returns True? (Scenario 9 — this was the bypass bug)
- Does seeding use `hash(unit_id)` derivation rather than sequential integers? (Scenario 7)
- Does `SeededRandom` use its own `_rng` instance rather than the global `random` module? (Isolation)
- Does the loop expression (`loop_until`) have a `max_iterations` guard to prevent infinite loops?
- Are underscore-prefixed variables (`_card_val`, `_internal`) handled consistently?

**Why:** The expression step validation bypass (Scenario 9) let 456 mathematically incorrect Blackjack hands through. The drunken sailor seeding bug (Scenario 7) produced 77.5% bias.

### Focus Area 4: Provider and Cost Handling

**Where:** `format_step_provider_tag()` (~line 1157), `compute_step_cost()` (~line 1140), `check_prerequisites()`, `scripts/tui/screens/main_screen.py` — `_calculate_cost_from_manifest()`

**What to check:**
- `format_step_provider_tag()`: is there a None guard for `provider_instance`? (BUG-4 — currently crashes)
- `_calculate_cost_from_manifest()` in `main_screen.py`: does it load pricing from `models.yaml` or fall back to hardcoded rates? (BUG-6 — currently uses hardcoded `output_rate=0.30`)
- `check_prerequisites()`: does it correctly enumerate ALL providers needed (global + per-step)?
- Rate limit backoff: is there exponential backoff for 429 responses in batch mode? (BUG-5 — currently tight retry loop)

**Why:** BUG-6 caused a $1,856 run to appear to cost $109 (17x undercount). BUG-4 crashes the orchestrator whenever the API key is missing.

### Focus Area 5: Fan-Out Step Handling (`scripts/orchestrate.py`)

**Where:** `run_fan_out_step()`, `verify_run()` fan-out branch, `get_next_chunk_number()`

**What to check:**
- Are child IDs correctly formatted as `{parent_id}__fan{NNN}` (zero-padded 3 digits)?
- Is `_fan_parent_id` and `_fan_index` added to each child record?
- After fan-out, is the parent chunk correctly marked as `VALIDATED` (terminal)?
- Does `verify_run()` skip fan-out steps (different ID space) without counting children as orphans?
- Does `get_next_chunk_number()` correctly handle the case where fan-out creates many new chunks?

**Why:** Fan-out steps create a new unit namespace. Incorrect child ID generation or verification can produce false "missing" reports or corrupt the unit lineage.

### Focus Area 6: TUI Status Guards and State Machine (`scripts/tui/screens/home_screen.py`)

**Where:** Resume handler (R key), Pause handler (P key), Kill handler, status-based guards

**What to check:**
- Does the resume handler allow "pending" status? (BUG-3 — currently excluded)
- Do pause and kill handlers allow "stuck" status? (BUG-2 — currently excluded)
- Does the resume handler distinguish "pending" (needs `--watch`) from "paused" (needs re-attach)?
- Is the restart dialog message accurate? (BUG-1 — currently implies data loss)

**Why:** These state machine gaps leave users unable to manage their runs. BUG-2 means stuck (rate-limited) runs cannot be stopped. BUG-3 means freshly initialized runs require CLI workarounds.

### Focus Area 7: Missing Safeguards (Pre-Run Commitment)

**Where:** `init_run()` (~line 3052), `scripts/tui/screens/new_run_modal.py`

**What to check:**
- Does `init_run()` show estimated cost before creating the run? (BUG-7)
- For fan-out pipelines, is there any estimate of the fan-out multiplier?
- Is there a `--yes` flag that gates confirmation?
- Does the TUI new run modal show cost estimates before the Launch button?

**Why:** BUG-7: A fan-out run expanded from 5,281 to 51,712 units with no warning. Estimated cost ~$100, actual cost ~$1,856.

## Guardrails

- **Line numbers are mandatory.** If you cannot cite a specific line number, do not include the finding. Use `grep -n 'pattern' file` or `read file --offset N` to find exact lines.
- **Read function bodies, not just signatures.** Don't assume a function works based on its name. Read the actual implementation.
- **If unsure whether something is a bug or a design decision**, flag it as QUESTION, not BUG. Check `ai_context/DEVELOPMENT_CONTEXT.md` for known design decisions and technical learnings.
- **Grep before claiming missing.** If you think a feature is absent, search the codebase. If found in a different file, that's a location defect (PHANTOM), not a missing feature.
- **Do NOT suggest style changes, refactors, or improvements.** Only flag things that are incorrect, could cause failures, or diverge from documented requirements.
- **Known bugs are in DEVELOPMENT_CONTEXT.md.** Do not re-report BUG-1 through BUG-7 as new findings unless you've found a different instance or root cause.

## Phase 2: Regression Tests for Confirmed BUGs

After producing BUG findings, write regression tests to `quality/test_regression.py`.

Each test should:
1. Target the exact code path and line numbers from the finding
2. Fail on the current implementation (confirming the bug)
3. Include the finding description in the docstring for traceability

Save a confirmation table in the review file:

```
| Finding | Test | Result | Confirmed? |
|---------|------|--------|------------|
| BUG-4: format_step_provider_tag crashes on None | test_scenario_13_... | XFAIL | YES — bug confirmed |
| BUG-6: hardcoded pricing fallback | test_scenario_14_... | XFAIL | YES — bug confirmed |
```

**Note:** Tests for BUG-4 and BUG-6 already exist in `quality/test_functional.py` as `xfail` tests
and have confirmed those bugs. Do not duplicate them — reference them in the confirmation table.
