## Work Package 1: CLI Foundations

### 1A: --restart flag

**What:** Find the running PID for a run, send SIGTERM, wait up to 5 seconds, escalate to SIGKILL if needed, then relaunch the orchestrator with the same arguments.

**CLI:** `python3 scripts/orchestrate.py --restart --run-dir <path>`

**Implementation:**
1. Read `orchestrator.pid` from the run directory
2. Check if the process is alive using `os.kill(pid, 0)`
3. Send SIGTERM, wait up to 5 seconds for exit
4. If still alive, send SIGKILL
5. Read the manifest to determine the original mode (batch/realtime) and provider
6. Relaunch with `--watch` (batch) or `--realtime` on the same `--run-dir`

**Where to look:** `scripts/orchestrate.py` argparse section (~line 6200+) for the flag. The kill logic can reference `run_tools.py` or be inline.

**Tests:**
- Test: SIGTERM sent to running PID
- Test: SIGKILL sent after timeout
- Test: relaunches with correct mode
- Test: handles already-dead PID gracefully

### 1B: Expression step batch mode optimization

**What:** Move expression step detection before the `max_inflight_batches` throttle so local CPU operations don't wait for API rate limit slots.

**Implementation:** In `tick_run()` or wherever batch submission is throttled by `max_inflight_batches`, check if the current step has `scope: expression` BEFORE checking the inflight limit. Expression steps run locally and don't consume API slots.

**Where to look:** `scripts/orchestrate.py` — search for `max_inflight` or the batch throttle logic.

**Tests:**
- Test: expression step executes immediately even when max_inflight_batches is reached
- Test: LLM steps still respect the inflight limit

### 1C: Named runs

**What:** Allow users to assign a human-readable name to runs. Stored in `manifest.metadata.display_name`.

**CLI:** `python3 scripts/orchestrate.py --name "Haiku 1k baseline" --run-dir <path>` (sets the name on an existing run)

Also accept `--name` during `--init` to set the name at creation time.

**Implementation:**
1. Add `--name` argparse argument
2. If used with `--init`, store in manifest metadata during initialization
3. If used alone with `--run-dir`, update the manifest metadata on an existing run
4. The TUI home screen should display this name (implemented in Work Package 3)

**Where to look:** `scripts/orchestrate.py` argparse section and the `init_run()` function.

**Tests:**
- Test: `--name` during `--init` stores name in manifest
- Test: `--name` on existing run updates manifest
- Test: `--name` without `--run-dir` errors gracefully

### 1D: Pipeline report (--report CLI)

**What:** Rich post-run (or mid-run) report showing validation funnel, failures by item, top errors by step, cost summary, and post-processing results. Short run names resolve automatically (e.g., `bj_haiku_1k` → `runs/bj_haiku_1k`).

**CLI:** `python3 scripts/orchestrate.py --report <run_name>`

**Implementation:**
Create a `generate_report()` function (in `scripts/run_tools.py` or a new `scripts/report.py`) that:

1. Resolves short run names: if `<run_name>` doesn't start with `runs/`, prepend it. If that path doesn't exist, search `runs/` for a matching prefix.
2. Reads the manifest for run metadata (provider, model, mode, status, timing).
3. Walks all chunks across all steps to build the validation funnel:
   - Per step: valid count, failed count, cumulative lost, pass %, yield %
   - Reads both `.jsonl` and `.jsonl.gz` files (use the transparent loading from v1.0.3)
4. Groups failures by item (parsed from unit_id prefix before `__rep`).
5. Extracts top 3 error messages per step from failure records (use `verification_details`, `strategy_verification_details`, or `error` fields).
6. Calculates cost using model registry pricing with correct batch/realtime multiplier.
7. Includes post-processing output files if they exist (strategy_comparison.txt, etc.).
8. Outputs to stdout as a formatted report.

**Tests:**

## Work Package 2: Analysis Tooling

### 2A: Cross-run comparison

**What:** Compare strategy results, failure rates, and pass rates across multiple runs side by side.

**CLI:** `python3 scripts/orchestrate.py --compare run1 run2 run3`

Or as a standalone script: `python3 scripts/compare_runs.py run1 run2 run3`

**Implementation:**
1. Accept 2+ run names (short names resolve like `--report`).
2. For each run, extract: provider/model, total units, valid units, pass rate, cost, and the post-processing results (strategy_comparison.txt or equivalent).
3. Generate a side-by-side comparison table.
4. Output both to stdout AND save as a markdown file (e.g., `comparison_20260308_143000.md` in the current directory).

**Output format:**

**Tests:**
- Test: comparison with 2 runs
- Test: comparison with 3+ runs
- Test: handles runs with different step counts gracefully
- Test: markdown output file is created

### 2B: Failure breakdown by field

**What:** Group failures by any field, showing differential failure rates across categories.

**CLI:** `python3 scripts/orchestrate.py --report <run_name> --failures-by <field_name>`

Or integrated into the `--report` output when failure records contain the specified field.

**Implementation:**
1. Read all failure records across all steps.
2. Group by the specified field (e.g., `strategy_name`, `difficulty`, or any field present in the failure records — which now include upstream fields thanks to v1.0.3 Bug 3 fix).
3. Show counts per step per group value.

This is essentially the "FAILURES BY ITEM" section of the pipeline report, but parameterized on any field instead of hardcoded to unit_id prefix.

**Tests:**
- Test: grouping by a field present in failure records
- Test: grouping by a field not present falls back gracefully
- Test: works with upstream fields (strategy_name from items file)

### 2C: Hand-by-hand diff across runs

**What:** Given two runs with identical seeds (same pipeline, same repeat count), show how different models played the same dealt hand. Highlights where models diverge on identical input.

**CLI:** `python3 scripts/compare_hands.py run1 run2 [--unit-id <id>] [--sample N]`

**Implementation:**
1. Load validated output from the play_hand step (or equivalent LLM step) for both runs.
2. Match units by `unit_id` — identical seeds mean identical deals.
3. For each matched pair, compare: `result`, `player_final_total`, `dealer_final_total`, `first_action`, `player_busted`, `dealer_busted`.
4. Flag divergences — cases where the same cards produced different outcomes.
5. `--unit-id` shows full action log comparison for one specific unit.
6. `--sample N` shows N random divergent pairs.

**Tests:**
- Test: matching units by unit_id across runs
- Test: detecting divergences in results
- Test: handling runs with different valid sets (some units failed in one run but not the other)

## Work Package 3: TUI Integration

Wire all CLI features from Work Packages 1 and 2 into TUI screens and key bindings. Each feature calls the same underlying function as the CLI — the TUI is a presentation layer only.

### 3A: Pipeline report modal (G key)

**What:** Press G on the detail screen to generate and display the pipeline report in a scrollable modal.

**Implementation:**
1. Call the same `generate_report()` function from Work Package 1D.
2. Display in a `ModalScreen` with scrollable content.
3. Include a "Copy to clipboard" button (C key in the modal).

### 3B: Named runs display (N key)

**What:** Press N on home screen with a run selected to set or edit the display name. Display the name in the home screen table.

**Implementation:**
1. N key opens a text input modal (use existing modal patterns).
2. On submit, writes to `manifest.metadata.display_name`.
3. Home screen table shows the display name in a new column (or replaces the run directory name if a display name exists).

### 3C: Switch batch↔realtime (M key)

**What:** Press M on detail screen for a paused run to schedule a mode switch. The orchestrator waits for outstanding batch chunks to return, then flips to the new mode for remaining work.

**Implementation:**
1. M key only enabled for paused runs.
2. Shows confirmation modal: "Switch from batch to realtime? Outstanding batch chunks will complete first."
3. On confirm, writes `metadata.scheduled_mode_switch: "realtime"` (or "batch") to manifest.
4. When the orchestrator resumes (R key or `--watch`), it reads the scheduled switch and acts accordingly.
5. The orchestrator logic: if `scheduled_mode_switch` is set, poll for any outstanding batches until they complete, then switch to the new mode for remaining steps.

### 3D: Intermediate results (P key)

**What:** Press P on detail screen for an in-progress run to run post-processing on current data.

**Implementation:**
1. Calls the real post-processing scripts from config.yaml.
2. Output files are timestamped: `strategy_comparison_20260308_143000.txt` (not overwriting the eventual final output).
3. Results displayed in a modal after generation.
4. Also works on completed runs (regenerates with timestamp).

### 3E: AI-assisted troubleshooting (T key)

**What:** Press T on detail screen when failures exist. Generates a troubleshooting prompt, lets user preview and edit, sends to LLM, displays response.

**Implementation:**
1. T key only enabled when the run has failures.
2. Generates prompt from a global template at `scripts/templates/troubleshoot.jinja2` containing:
   - Pipeline name, provider, model
   - Failure summary (count by step, top error messages)
   - 3 sample failed units with full context
   - Config snapshot (relevant sections)
   - Instructions for the LLM: "You are an Octobatch expert. Diagnose why these units are failing and suggest fixes."
3. Shows prompt in a preview modal. User can edit if needed.
4. User selects provider (default: cheapest available with configured API key). Provider picker shows estimated cost.
5. Sends prompt via realtime API call.
6. Displays response in a scrollable modal with "Copy to clipboard" option.

### 3F: --restart TUI integration (R key enhancement)

**What:** Extend the R key on home screen. If the run is paused/detached, resume (existing behavior). If the run is running, offer to restart (kill + relaunch).

**Implementation:**
1. If run status is "running" and process is alive: show confirmation "Run is active. Restart? This will kill the current process and relaunch."
2. On confirm: call the same kill + relaunch logic from Work Package 1A.
3. If run status is paused/detached: existing resume behavior (unchanged).

### 3G: Cross-run comparison (C key)

**What:** Select multiple runs on home screen, press C to compare.

**Implementation:**
1. Add multi-select capability to home screen (e.g., Space to toggle selection, highlighted rows).
2. C key generates comparison using the same function from Work Package 2A.
3. Display in a modal with "Copy to clipboard" option.

### General TUI notes:
- All modals should be scrollable for long content.
- All modals should support C key for "Copy to clipboard" using existing clipboard infrastructure.
- Key bindings should be shown in the footer bar when available.
- Keys should be disabled/hidden when not applicable (e.g., T only when failures exist, M only when paused).

## Work Package 4: Framework Features

### 4A: Fan-out steps

**What:** New `scope: fan_out` step type that explodes an array field from parent units into individual child units.

**Config:**
```yaml
pipeline:
  steps:
    - name: generate_variants
      prompt_template: generate.jinja2
    - name: fan_out_variants
      scope: fan_out
      field: variants        # array field to split on
      child_field: variant   # name for the individual element in child units
    - name: evaluate_variant
      prompt_template: evaluate.jinja2
```

**Implementation:**

1. **New step type detection:** In the orchestrator, detect `scope: fan_out` steps like expression steps. Fan-out steps are exempt from the 4-Point Link Rule (no template, schema, or validation entry needed).

2. **Fan-out execution:** When a fan-out step runs:
   a. Read the previous step's validated output for the chunk.
   b. For each unit, read the array field specified by `field`.
   c. Create child units: one per array element. Each child gets:
      - `unit_id`: `{parent_unit_id}__fan{NNN}` (zero-padded index)
      - All parent fields (inherited)
      - `{child_field}`: the individual array element
      - `_fan_index`: integer index within the parent's array
      - `_fan_parent_id`: the parent's unit_id
   d. Pack child units into new chunks respecting `chunk_size`. Never modify existing chunks — append new ones with sequential IDs.
   e. Write the fan-out output as `{step}_validated.jsonl` (fan-out has no failure mode — it's deterministic expansion).
   f. Update the manifest with the new chunks.

3. **Chunk management:** New chunks created by fan-out follow the same pattern as retry chunks. The manifest discovers them on the next tick. Sequential chunk IDs continue from the highest existing ID.

4. **Downstream steps:** Process the fanned-out chunks normally. Each child unit flows through remaining steps independently.

5. **Retry interaction:** If a parent unit fails and retries in a prior step, the retry's fan-out creates children in new chunks at the end. No duplication because the original failed parent never produced children.

6. **--verify support:** When walking the pipeline and encountering a fan-out step, read the fan-out output to determine the expected downstream count (sum of array lengths across valid parents) instead of carrying forward the parent count.

7. **Pipeline report:** Show fan-out boundary in the validation funnel:
   ```
   generate_variants       500 valid / 500 total  (100.0%)
   fan_out_variants       2350 units created from 500 parents
   evaluate_variant       2300 valid / 2350 total  (97.9%)
   ```

**Where to look:**
- Expression step handling in `orchestrate.py` for the pattern to follow
- `generate_units.py` for how initial chunks are created (similar logic for fan-out chunk creation)
- `run_tools.py` for `--verify` modifications

**Tests:**
- Test: fan-out creates correct number of child units
- Test: child unit IDs follow `{parent}__fan{NNN}` format
- Test: child units inherit parent fields
- Test: child units contain the individual array element in the specified field
- Test: chunk_size is respected — new chunks created when full
- Test: fan-out with empty array produces 0 children for that parent
- Test: fan-out with retry — late parent creates children in new chunk
- Test: --verify correctly reports fan-out count discontinuity
- Test: downstream LLM step processes fanned-out units normally

### 4B: Re-validate failures

**What:** Re-validate raw LLM responses against updated schemas/validation rules without re-calling the API. Saves cost when you fix a schema or validation rule bug.

**CLI:** `python3 scripts/orchestrate.py --revalidate --run-dir <path> [--step <step_name>]`

**Implementation:**
1. For each failed unit in the specified step (or all steps if no step specified):
   a. Read the raw LLM response from the failures file or the raw results file.
   b. Re-run schema validation and business logic validation using the CURRENT pipeline config (not the snapshot — the user may have updated schemas/rules).
   c. If the unit now passes: move it from failures to validated output.
   d. If it still fails: leave it in failures with updated error message.
2. Update the manifest counts.
3. If any units were recovered, offer to re-run post-processing.

**Where to look:**
- `scripts/run_tools.py` for the function
- `scripts/schema_validator.py` and `scripts/validator.py` for the validation pipeline
- The validation pipeline in `orchestrate.py` (`run_validation_pipeline()`) for the pattern

**Tests:**
- Test: unit that failed under old rules passes under updated rules → moved to validated
- Test: unit that still fails → stays in failures with new error
- Test: manifest counts updated after revalidation
- Test: only specified step is revalidated when `--step` is used

### 4C: AI-driven stochastic integration testing

**What:** Run QUALITY.md fitness scenarios N times. Aggregate pass rate is the metric. Failures distilled into scripted regression tests.

**Implementation:**
1. Create `tests/stochastic_runner.py` that:
   a. Reads QUALITY.md and extracts the 10 fitness scenarios.
   b. For each scenario, runs the verification N times (default N=5).
   c. Records pass/fail for each run.
   d. Reports aggregate pass rates per scenario.
   e. For any scenario that fails at least once: extract the failure details and generate a minimal pytest test that reproduces the failure condition.
2. Output: a summary table + any generated regression tests written to `tests/generated/`.

**CLI:** `python3 tests/stochastic_runner.py [--runs N] [--scenario S]`

**Tests:**
- Test: runner correctly parses QUALITY.md scenarios
- Test: runner handles scenario pass and fail
- Test: generated regression tests are valid pytest
